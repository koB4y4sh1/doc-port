Article Summary

VS Code は AI を「コード生成」だけでなく、開発運用全体に使っている

この記事の本質は、VS Code チームが AI エージェントを使って 開発速度を上げただけでなく、リリース運用・課題整理・レビュー・検証まで含めた 開発フロー全体を作り替えた、という点です。

結論: VS Code チームは AI により、10年間続けた月次リリースから週次リリースへ移行しました。 ただし、単純に速くしたのではなく、 「並列実行」「中間ドキュメント削減」「運用自動化」「品質ハーネス」「人間の最終判断」 をセットで整備したことが重要です。

1. 何が変わったのか

以前の VS Code は、長年にわたり月次リリースのリズムで運用されていました。 しかし AI エージェント活用により、品質基準を保ったまま週次リリースへ移行しています。

従来: 月次で計画 → 実装 → テスト → エンドゲーム → 出荷
現在: AIで並列実行 → 短サイクル検証 → 週次リリース
ポイント: リリース頻度の向上そのものが目的ではなく、 バグ修正や新機能をユーザーへ数日単位で届けられるようにしたことが狙いです。

2. 記事で示された6つの学び

  1. 自分を並列化する: 会議前に複数のエージェント作業を起動し、待ち時間を減らす。
  2. 中間成果物を減らす: 会議メモ → Issue → 仕様書 → 実装、という段階を短縮する。
  3. 速度増加で膨らむ雑務を自動化する: 課題整理、コミット要約、リリースノート、コードレビューを AI で支援する。
  4. 速度より先に品質ハーネスを作る: テスト、Golden Scenario、レビューゲートを先に整備する。
  5. 責任の持ち方を再定義する: PM や他領域の開発者も貢献しやすくなるが、最終責任はエンジニアが持つ。
  6. 最後の“良さ”は人間が判断する: AI は正しさを見やすいが、体験価値やプロダクトの気持ちよさは人間が見る。

3. 実務での使い方: 並列で仕事を進める

記事では、会議に入る前に 3〜4 本のエージェント作業を起動し、 会議中も別セッション・worktree・クラウド上で並列実行する運用が紹介されています。

  • バグ修正を複数並列で投げる
  • 機能プロトタイプを先に作らせる
  • Issue のトリアージを自動で回す
  • 会議後に出力をレビューして再指示する
重要: これは AI に丸投げしているのではなく、 人が「何を任せるか」「どれを採用するか」を高速に判断するための運用です。

4. 中間ドキュメントより PR を重視

従来の PM フローは、仕様書や PRD を作り、Issue を切って、実装を依頼する流れでした。 しかし記事では、PM 自身が AI エージェントを使ってプロトタイプ PR を作る流れが強調されています。

従来 → 仕様書 → Issue → 開発依頼
現在 → AIで試作 → PR作成 → 設計レビュー

つまり、文書で仮説を説明するより、 動く PR を出して会話を前に進める方が速い、という考え方です。

5. 自動化している運用領域

VS Code チームは、実装以外の周辺業務も AI で自動化しています。

  • コミット要約: 過去24時間の複数リポジトリの変更を収集し、要約する。
  • リリースノート作成: 変更要約を Insiders の変更履歴や日次更新に流用する。
  • Issue トリアージ: 重複検知、担当推定、ラベル候補付与を自動実行する。
  • GitHub 上の補助 UI: 重複候補や担当候補を GitHub 画面上で見られるようにする。
  • エディタ内コマンド: VS Code 内から Issue 整理や重複確認を行えるようにする。

7. 品質を落とさないための仕組み

  • 自動検証: Playwright MCP を使うエージェントが UI を開き、スクリーンショットを取り、期待動作と比較する。
  • テスト強化: 単体・結合テストに加え、重要なユーザーフローを Golden Scenario として定義する。
  • マージ後検証: これまで人がやっていた確認を、エージェントにも走らせる。
  • 自動コードレビュー: Copilot のレビュー結果を人間レビュー前に解消する。

8. それでも人間が残る理由

記事で繰り返し強調されているのは、 AI が見つけやすいのは「正しいかどうか」であり、 人間が見るべきなのは「本当に気持ちよい体験かどうか」という点です。

AIの役割: 重複検知、要約、コードレビュー、検証、自動修正
人間の役割: アーキテクチャ判断、品質責任、体験価値の判断

9. この記事から学べる実務上の示唆

  1. AI 導入の本質はコード生成ではなく、開発プロセス全体の再設計にある。
  2. AI で速度を上げる前に、テスト・検証・レビューの仕組みを先に作るべき。
  3. 仕様書を厚くするより、試作 PR を早く出して議論する方が学習速度は高い。
  4. PM やマネージャーも、エージェントにより実装プロセスへ直接参加しやすくなる。
  5. 最終的な責任移譲は起きておらず、責任の所在は依然として人間側にある。

10. ひとことで説明するなら

VS Code チームは、AI を「補助ツール」としてではなく、 開発・運用・品質保証を再構成するための実行基盤として使っている。
